# Active Plan: Battle Mode, Users, Leaderboard, Admin

Подробные требования и контекст: [/home/yahor/telegram-miniapp-demo/docs/concept_details.md](/home/yahor/telegram-miniapp-demo/docs/concept_details.md)

## 1) Конфиг и интервалы

- Добавить конфиг-переменные частоты системных логов.
- Добавить конфиг-переменные частоты битв.
- Подключить эти интервалы к runtime-триггерам (таймеры/циклы).

## 2) Сущность пользователя

- Ввести сущность `User` (пока локальная модель).
- Связать `User` с профилем персонажа и его состоянием (`lvl`, `HP`, `ATTACK`, цвет, вещи).
- Подготовить точку расширения под Telegram-данные пользователя в будущем.

## 3) Боевая модель

- Реализовать расчет боевой силы по согласованной формуле `power`.
- Реализовать перевод силы пары в вероятности (`pA`, `pB`) и выбор исхода боя.
- Связать исход боя с реакцией системы (логи, реплики, изменение состояния).

## 4) Матчмейкинг

- Реализовать `mmScore` на базе `power`.
- На каждом боевом тике: перемешивание, сортировка по `mmScore`, паринг соседей.
- Добавить порог `maxDelta` для фильтра неравных пар.
- Обработать случай непарного участника (пропуск тика).

## 5) Пост-бой: вещи

- Для победителя: сначала построение пула улучшений, затем шанс дропа.
- Если пул улучшений пуст, дроп не выполняется.
- Выбор вещи делать весами редкости (более сильные вещи выпадают реже).
- Для проигравшего: шанс даунгрейда с вероятностью, зависящей от `delta` (сильный даунгрейд маловероятен).
- После изменения вещи пересчитывать `HP` и `ATTACK`.

## 6) Пост-бой: цвет

- После победы: шанс апгрейда цвета только на 1 ступень, без даунгрейда.
- После поражения: шанс даунгрейда цвета только на 1 ступень, без апгрейда.
- Убрать любые перерандомы цвета через полный пересчет распределения после боя.

## 7) Логи и триггеры

- Соединить уже разделенные типы логов с реальными триггерами системы.
- Системные логи публиковать по конфиг-интервалу.
- Боевые логи публиковать по факту завершения боев и их исхода.
- Логи дропа/обновления публиковать по факту изменений инвентаря/уровня.

## 8) Реплики персонажа

- Привязать реплику к типу события и исходу боя (победа/поражение).
- Обеспечить вызов реплики в едином пайплайне вместе с логом события.

## 9) Лидерборд

- Реализовать расчет `rating` на той же базе, что и боевая оценка (`lvl`, `HP`, `ATTACK`).
- Построить сортировку и отображение топа пользователей.
- Обновлять топ при изменениях состояния пользователей.

## 10) Админка

- Добавить вход в админку отдельной кнопкой рядом с кнопкой профилей.
- В админке показать лидерборд.
- Добавить кнопку завершения игры.
- При завершении: заморозить лидерборд и отправить системный лог всем пользователям с победителем.

## 11) Viewprofile и список пользователей

- Добавить кнопку «добавить пользователя» на `viewprofile`.
- Добавлять пользователей как профили в список.
- Отделить секцию пользователей визуальной линией после профилей.

## 12) Порядок реализации

- Этап A: конфиг-интервалы, сущность пользователя, базовые триггеры логов.
- Этап B: матчмейкинг + боевая модель + пост-бой изменения.
- Этап C: лидерборд + админка + завершение игры.
- Этап D: интеграция UI-потока (`viewprofile`, добавление пользователей, визуальные разделители).
