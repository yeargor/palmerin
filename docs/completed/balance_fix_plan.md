# Balance Fix Plan (Telemetry-Driven)

Этот документ фиксирует текущий анализ проблем баланса по telemetry и задает стратегию исправлений + как возобновлять анализ после новых симуляций.

Источник процесса и полей telemetry: `docs/telemetry_framework.md`.

## Baseline Наблюдения (пример прогона)

Пример, на котором основан анализ (для ориентира; после фиксов цифры могут измениться):
- Симуляция: `--users 8 --duration-ms 120000`
- События: `35` (`34 battle_tick` + `1 game_finished`)
- Боёв: `113`
- Upset rate: `36.3%` (победа андердога по win-prob)
- Matchmaking decisions: `accepted=113`, `rejected=125` (reject `52.5%`)
- Reject reasons:
  - `already_paired=99` (≈`79%` отказов)
  - `delta_exceeded=26` (≈`21%` отказов)
- Participation (боёв на игрока): `min=13`, `max=34`, spread `21`
- Unmatched: `46` суммарно, `23/34` тиков, среднее `1.35` на тик
- Лидерборд: top-1 сменился `13` раз; был камбэк (`u2` с `8` места до `1`)
- При этом был “застрявший” игрок: пример `u4` с WR ≈`15%`, финальный `lvl=3` при лидерах `lvl=19`

## Критичные проблемы (по важности)

### 1) Неравная participation ломает баланс

Факт: в baseline spread по числу боёв был очень большим (`13..34`).

Почему это критично:
- Победитель гарантированно получает `lvl+1`; кто чаще участвует, тот быстрее растет.
- Система апгрейдов/даунгрейдов (вещи/цвет) также “интегрируется” по числу боёв: больше боёв = больше шансов улучшения.
- Игрок с низкой participation может застрять в слабом состоянии, даже если формула боя “в целом ок”.

Стратегия: сначала добиться близкого числа боёв на игрока за фиксированное время. Иначе любые правки “вероятностей дропа” будут маскировать проблему.

### 2) Matchmaking теряет пары из-за `already_paired`

Факт: большинство отказов в decisions были `already_paired` (≈`79%` всех reject).

Почему это критично:
- Это не “баланс сил”, а артефакт алгоритма подбора пар в тике.
- Это напрямую ухудшает participation и качество матчей (кто-то чаще остаётся unmatched).

Стратегия: подбирать пары так, чтобы максимизировать число сыгранных матчей внутри порога delta.
Практически: рассматривать тиковый matchmaking как задачу паросочетания, где цель:
- максимизировать количество пар `withinDelta`;
- вторичная цель: минимизировать суммарную `delta` внутри принятых пар.

### 3) Snowball/anti-snowball в прогрессии (уровень/вещи/цвет)

Наблюдение: апсеты были достаточно высокими (это хорошо для “игровости”), лидер менялся часто (камбэки возможны).
Но: всё равно победы сильно коррелируют с финальным местом; “слабый” игрок может деградировать/застревать.

Стратегия: не трогая формулу исхода боя, сделать прогрессию зависимой от expected outcome:
- За ожидаемую победу награда меньше, за апсет больше (underdog-bonus).
- Для проигравшего добавить “страховку” от серии поражений (уменьшать штрафы после N лузов подряд и/или ограничивать потерю ключевых слотов).

## Порядок фиксов (как работать, чтобы не потерять причинность)

1. Исправить participation (цель: сузить spread по `fights` между игроками).
2. Исправить matchmaking (уменьшить `already_paired` rejects, увеличить долю `accepted`).
3. Затем настраивать прогрессию (lvl/loot/color) и анти-snowball.

Важно: не менять сразу 2-3 вещи, иначе telemetry не покажет, какой фикс дал эффект.

## Метрики, которые нужно сравнивать “до/после” после каждого фикса

### Participation / fairness
- `minF/maxF/avgF/spread` по `fights` на игрока.
- `unmatchedTotal`, `unmatchedTicks`, `avgUnmatchedPerTick`.

Целевые сигналы улучшения:
- spread значительно уменьшается.
- unmatched уменьшается.

### Matchmaking качество
- `rejectedShare` (доля reject от всех decisions).
- breakdown reject reasons: доля `already_paired` должна падать.
- `avgAcceptedDelta` и распределение delta в accepted (не должно резко расти).
- `pairsPerTick` должно расти (при сохранении разумной delta).

### Баланс “игровости”
- `upsetRate` (не должен стать слишком низким).
- `leaderboard topChanges` (не должен стать нулём; иначе получится “замороженный” лидер).
- Корреляции (по желанию): wins vs final rank; fights vs wins.

### Power drift / snowball
- drift `power` и `level` по игрокам.
- наличие “stuck игроков” (очень низкий lvl при высоком у лидеров после одинакового времени).

### Транзакции вещей/цвета
- Winner/Loser: сколько `itemChanged`, `colorChanges`, ап/даун по tier.
- Проверка, что ап/даун происходит только на 1 tier (если так задумано).

## Как возобновить анализ после фиксов (быстрый чеклист)

Опираться на `docs/telemetry_framework.md`.

1. Поднять telemetry-сервер: `npm run serve:logs`
2. Smoke: `npm run simulate:battle:smoke`
3. Полный прогон:
   - `node scripts/simulate-battle-run.mjs --users 8 --duration-ms 120000`
4. Базовый анализ: `npm run analyze:telemetry`
5. Deep-dive по `artifacts/battle/battle-events.jsonl`:
   - participation spread, unmatched
   - reject reasons (особенно `already_paired`)
   - upsetRate
   - power/level drift
   - item/color transitions
   - RNG trace sanity (means ~0.5, min~0, max~1)

Если результат “странный”:
- проверить, что используется `npm run serve:logs`, а не `python -m http.server` (иначе telemetry в файл не пишется).
- не смешивать прогоны: симуляция должна очищать JSONL в начале; фильтровать по `runId`, если в файле несколько прогонов.
