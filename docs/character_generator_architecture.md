# Character Generator Architecture

## 1) Назначение
Этот документ описывает текущую архитектуру генератора ASCII-персонажей в UI, правила работы с компонентами и выравниванием, а также практические ограничения, чтобы при дальнейшем развитии не ломать геометрию, высоту и центрирование.

Канонические исходники форм/идей:

## 2) Базовые инварианты
- Персонаж всегда собирается из 5 слотов:
  - `hat`
  - `face`
  - `arms`
  - `torso`
  - `legs`
- Слот нельзя подменять чужим компонентом (`slot`-валидация обязательна).
- Рендер идет на единую сетку:
  - `SPRITE_GRID_WIDTH = 24`
  - `SPRITE_MIN_GRID_HEIGHT = 7` (фиксированная высота)
- Нижнее выравнивание фиксировано:
  - `SPRITE_BOTTOM_PADDING_ROWS = 1`
  - нижний заполненный ряд должен быть на `height - 2`
- Глобальное правило для стабильности UI:
  - форма/позиция контента внутри грида может меняться только по правилам компонентов,
  - сам контейнер спрайта не должен “прыгать” по высоте.

## 3) Модель компонента
Компонент описывается объектом `SpriteComponent`:
- `id`: уникальный идентификатор компонента.
- `slot`: слот принадлежности (`hat|face|arms|torso|legs`).
- `layers[]`: набор слоев рендера.
- `effects[]`: эффекты, влияющие на якоря/сборку (например, сдвиг слотов вниз).
- `constraints`: `requires/forbids` (сейчас почти не используются; минимальный набор).
- `anchor`: база центрирования компонента.

`layer`:
- `rowOffset`: вертикальный сдвиг от якорного ряда слота.
- `col`: базовая колонка слоя до центрирования.
- `text`: ASCII/Unicode-фрагмент.
- `className`: CSS-класс символов (`sprite-main`, `sprite-gun`, `sprite-dust`, и т.д.).
- `zIndex`: приоритет наложения.

## 4) Центрирование компонентов
### 4.1 Принцип
Центрирование выполняется не по левому краю и не по bbox всего спрайта, а по **основе компонента** (`anchor`).

`anchor` задается как:
- `pattern`: уникальный контекст (строка, которая ищется в одном из `layers[].text`).
- `targetIndexes`: диапазон индексов внутри `pattern` (0-based, включительно), который считается центром базы.

Пример:
- `pattern: "/|___|\\"`
- `targetIndexes: [2, 4]`
- Центр базы = середина от индексов `2..4`.

### 4.2 Почему так
- У многих компонентов есть “боковые хвосты” (звездочки, меч, длинные сегменты), которые не должны определять центр.
- Одинаковая визуальная посадка сохраняется даже при разной общей ширине слоя.

### 4.3 Важные правила для `anchor`
- `pattern` должен быть достаточно контекстным, чтобы не ловить ложные совпадения.
- Не использовать короткие неоднозначные паттерны вроде `"___"` без контекста.
- Лучше: полный контекст + диапазон внутри него.
- Если `pattern` не найден, код падает на fallback-центр по bounds; это допустимо как аварийный путь, но не как целевой режим.

## 5) Multi-row компоненты и оверлеи
Компонент может занимать несколько рядов через `layers` с разными `rowOffset`.

Это обязательно для:
- `arms_warrior` (две части меча на разных рядах),
- `arms_mage_mantle_top` (палочка + звездочки на нескольких рядах),
- `hat_mage` (двухрядная шляпа).

Правило:
- Центрируется только **одна база** компонента (`anchor`),
- все остальные слои этого компонента “едут” вместе с ней (через общий `dx`).

Это гарантирует, что доп. детали не теряются и не разъезжаются отдельно.

## 6) Вертикальные эффекты и смещения
`effects` позволяет компоненту менять якоря слотов.

Ключевой кейс:
- `hat_mage` добавляет `shift-slot-anchors` для `face/arms/torso/legs` на `+1` по `Y`.

Смысл:
- компоненты ниже шляпы не исчезают,
- просто опускаются как единый стек.

## 7) Алгоритм сборки (текущий)
1. Валидация пресета по слотам.
2. Построение якорей слотов (`BASE_SLOT_ANCHOR_ROW` + `effects`).
3. Для каждого компонента:
   - вычисление локального центра по `anchor`,
   - расчет `dx` до целевого центра слота,
   - генерация операций рендера (`ops`) для всех символов.
4. Нормализация по верхней границе (чтобы не было отрицательных рядов).
5. Фиксированная высота грида `7`.
6. Прижим к низу с нижним отступом `1` строка.
7. Наложение по `zIndex`.
8. Сборка `html` и `plainText`.

## 8) Random генерация
- Random выбирает по одному компоненту из каждого slot-пула.
- Запрещены только базовые невалидные состояния:
  - отсутствующий слот,
  - неверный slot-binding,
  - явные `requires/forbids` (если добавлены).
- Class-lock ограничений быть не должно (например, “магические руки только с магом”).

## 9) Высота, grid и “прыжки”
Текущая договоренность:
- Высота грида фиксирована: `7`.
- Внутренний контент всегда align к низу с `1` пустой строкой снизу.
- Изменение комбинации не должно визуально сдвигать весь stage-блок.

Если когда-то потребуется высота > 7:
- менять это как отдельную версию спецификации,
- одновременно обновлять CSS `--sprite-grid-rows`,
- прогонять визуальные регрессии по всем профилям.

## 10) Как добавлять новый компонент
1. Определить `slot`.
2. Добавить `layers` с реальными строками.
3. Добавить надежный `anchor`:
   - `pattern` (контекст),
   - `targetIndexes` (база).
4. При необходимости добавить `effects` (например, сдвиг нижних слотов).
5. При необходимости добавить `constraints` (минимально, только по фактическим конфликтам).
6. Включить компонент в `randomPoolBySlot` (если нужен в random).
7. Добавить/обновить тесты.
8. Проверить в `debugSprite=1` на мобильном viewport.

## 11) Что чаще всего ломает рендер (грабли)
- Центрирование по bbox вместо базы.
- Неоднозначный `anchor.pattern`.
- Короткий `pattern`, встречающийся в нескольких местах.
- Изменение `rowOffset` без учета `hat_mage` эффекта.
- Раздельное “ручное” смещение частей одного компонента вместо единого anchor-сдвига.
- Возврат к динамической высоте без синхронизации с CSS.
- Появление пустых/скрытых ног в random, из-за чего визуально “ломается” масса персонажа.

- `?debugSprite=1`

Что проверять:
- `grid WxH` в debug-панели,
- `line widths`,
- фактические строки спрайта,
- позицию персонажа внутри красной сетки.
