# Random Constructor Spec (v2)

## 1) Статус и цель
- Да, реализацию можно делать сейчас: готов исправлять генератор, вводить TDD и закладывать расширяемую архитектуру конструктора.
- Цель: `random` должен собирать валидного персонажа из 5 слотов (`hat/face/arms/torso/legs`) по правилам компонентов, а не по хардкоду классов.

## 2) Что показали примеры из `docs/random_sprites.txt`
- Нужны кросс-классовые комбинации (например, шляпа мага + пистолеты ковбоя + торс воина).
- `torso_mage` трактуется как торс (мантия), а не как ноги.
- Есть оверлеи, влияющие на несколько рядов (меч, звездочки) и не являющиеся отдельным слотом.
- У шляпы мага 2 ряда, поэтому лицо и нижние слоты должны сдвигаться по якорю, а не “пропадать”.
- Высота канвы не должна быть фиксированной `5`; нужен минимум `6` и рост при необходимости.

## 3) Инварианты конструктора
- Всегда ровно 5 слотов: `hat`, `face`, `arms`, `torso`, `legs`.
- Каждый компонент имеет обязательный `slot`; компонент нельзя ставить в чужой слот.
- Нельзя терять слот из-за геометрии.
- Геометрия рендера и слот-модель независимы:
- Слоты: логика.
- Канва: визуальная укладка.

## 4) Модель компонента (расширяемая)
- `id`
- `slot`
- `layers[]`: `{ row, col, text, className, zIndex }` в относительных координатах
- `effects[]`:
- пример: `shiftAnchor(face, +1)` при высокой шляпе
- пример: `spawnOverlay(stars)` для рук мага
- `constraints`:
- `requires`: список обязательных признаков/компонентов
- `forbids`: список несовместимостей
- `tags`: для совместимости и рандом-правил (например `head_tall`, `weapon_magic`, `face_bandana`)

## 5) Якоря и геометрия
- Вводится единая система якорей частей: `hatAnchor`, `faceAnchor`, `armsAnchor`, `torsoAnchor`, `legsAnchor`.
- Компоненты рисуются относительно якоря своего слота.
- `hat` может менять положение `faceAnchor` через `effects`, но `face` слот остается обязательным.
- Канва:
- `minHeight = 6`
- `minWidth = 24` (текущий общий baseline)
- `autoGrowHeight = true` при выходе оверлеев за верх/низ

## 6) Правила random-генератора
- Генератор выбирает кандидатов по слотам.
- Далее проходит универсальную валидацию:
- slot-valid
- constraints-valid
- geometry-fit (после применения effects и оверлеев)
- Если провал: новая попытка, без специальных `if` под конкретный класс.
- Фолбэк возможен только на “валидный пресет по умолчанию”, но не на ломанный спрайт.

## 7) TDD (принципы, не комбинации)
- Unit: `slot`-валидация.
- Unit: `requires/forbids`.
- Unit: применение `effects` (например, `hat_mage` сдвигает `faceAnchor`).
- Unit: мульти-ряд оверлеев (меч/звезды) и `zIndex`.
- Unit: `minHeight=6` + `autoGrowHeight`.
- Property-style: много random-итераций подряд без невалидных пресетов.
- Regression: текущие `warrior/mage/cowboy` визуально не меняются относительно baseline.

## 8) Несостыковки, которые считаются решенными в этой версии
- “5 рядов всегда” -> заменено на “5 слотов всегда, ряды динамические (min 6+)”.
- “layout как жесткая группа” -> заменено на constraint-driven совместимость.
- “магический набор только целиком” -> заменено на правила компонентов и якорей.

## 9) Этапы реализации
- Шаг 1: вынести ядро конструктора в чистые функции (для тестов).
- Шаг 2: ввести anchors/effects/constraints в каталоге компонентов.
- Шаг 3: переписать random-генератор на валидацию по правилам.
- Шаг 4: добавить и прогнать TDD.
- Шаг 5: визуальная проверка в `debugSprite=1` на мобильном viewport.
