# Concept Details

Касательно `str`, `dex`, `luk`, `hp` — пока что планирую остановиться на `hp` и `str` (возможно переименую `str` в `attack` для интуитивного понимания пользователями). Планирую оставить только две характеристики из этих четырех, чтобы можно было много не думать и видеть только уровень силы атаки и уровень защиты.

По поводу `rating` — это будет место в лидерборде между всеми игроками. В начале это будет рандомное число из общего числа подключившихся игроков, потом будет реальное место, когда наступит режим битвы.

По `lvl` — пока не знаю, чем сделать этот параметр и от чего он будет зависеть, но оставить надо, это канон.

Из динамических элементов:

- Вещи: элементами будут предметы в правой и левой руке и шапки.
- По рукам: возможно стоит использовать сразу комбинацию для левой и правой руки как сет (проще менять, но меньше рандома в комбинациях) — это нужно обдумать в будущем.
- Названия: будет массив приставок для названий шмоток (прилагательные), массив приставок для имени класса пользователя, сами классы и базовый голый персонаж (без вещей для каждого класса).
- Сами классы пока не придуманы.
- Приставки нужны, чтобы названия казались уникальными в какой-то степени (типа: «ржавый меч», «колючий щит», или «пьяный воин», «басовый маг» и так далее).

- Реплика будет зависеть от класса и от последнего лога, как реакция на тип события.
- Типы событий (логов) потом объявим.

Логи:

- Логи делятся на 4 типа: `system`, `combat`, `drop`, `levelup`.
- Смешные логи убираем: их функцию берет на себя реплика персонажа.
- `system` — системные сообщения (например, статус арены/матча), цвет текста белый.
- `combat` — боевые сообщения (и победы, и поражения), выделяются красным цветом.
- `drop` — выпадение предмета, отображается отдельно.
- `levelup` — повышение уровня/приятное обновление инвентаря, отображается отдельно.
- Для `drop` и `levelup` используется приятный теплый акцентный цвет (янтарный/коричневый диапазон).

## Новые детали по режиму битвы и логам

- Системные логи будут появляться раз в какое-то установленное время.
- Логи дропа и битв будут триггерить настоящие битвы, которые реализуем в будущем.
- Битвы тоже будут происходить раз в какое-то время.
- Из всего пула игроков (их персонажей) выбираются пары.
- Один персонаж может участвовать только в одной паре.
- Если остается персонаж без пары — он не участвует в битве.
- В рамках пары сравниваются показатели `HP` и `ATTACK`, и нужно придумать, как определять победителя в этом сравнении.
- После сравнения есть победитель и проигравший.

Для победителя:

- Есть шанс на выпадение вещи (только такой, которая лучше по статистикам, чем его текущая вещь).
- Есть гарантированное повышение `lvl` на одно очко.
- Вещь не обязана выпадать: сначала определяется, получит ли игрок вещь в принципе, потом сравниваются варианты вещей с лучшими статами.
- Дополнительно: чем больше параметры вещи, тем она должна быть реже.
- После выпадения вещи запускается пересчет `HP` и `ATTACK`.
- После обновления `HP` и `ATTACK` триггерится шанс обновления цвета персонажа на новый, но без даунгрейда цвета по редкости.

Для проигравшего:

- С `lvl` ничего не происходит.
- По вещи действует обратный принцип: шанс потерять вещь и получить вместо этого вещь хуже по параметрам.
- Слишком сильный даунгрейд вещи должен быть маловероятен.
- Пересчет `HP` и `ATTACK` тоже запускается на основе вещей.
- Для цвета апгрейд невозможен.
- Даунгрейд цвета работает так, что близлежащие уровни редкости имеют некоторый, но маловероятный шанс.
- Далеко лежащие уровни редкости должны быть невозможны.
- Цвета пересчитываются только в пределах одного цвета назад/вперед, чтобы не усложнять.

- В зависимости от исхода битвы триггерится соответствующий лог и соответствующая реплика.
- Значение `lvl` относится к профилю.
- `rating` — это положение в топе среди всех персонажей в игре (подключенных пользователей).
- Нужна сущность «пользователь».
- На странице `viewprofile` нужна кнопка «добавить пользователя».
- Новые пользователи должны добавляться как профили в список.
- Для этого секцию пользователей стоит отделить визуальной линией после профилей.
- Нужны переменные в конфиге для настройки частоты системных логов.
- Нужны переменные в конфиге для настройки частоты битв.
- Место в топе (`rating`) должно определяться по тому же принципу, что и исход битвы: на основе `lvl`, `HP` и `ATTACK`.
- Нужна админка (вероятно отдельной кнопкой рядом с крестиком открытия профилей).
- В админке должен быть лидерборд.
- В админке должна быть кнопка завершения всей игры.
- Завершение игры: фриз лидерборда и вывод в системные логи всех пользователей сообщения, что битва завершена, победитель: такой-то пользователь.

## Согласованный вариант расчета исхода битвы

- Используем игровую формулу силы персонажа:
- `power = (ATTACK / (HP + k)) + (HP / (ATTACK + k)) + w_lvl * lvl`
- Где `k` — малое положительное число для стабилизации формулы (например, `k = 1`).
- `HP` и `ATTACK` имеют равную значимость в формуле.
- `w_lvl` — отдельный коэффициент влияния уровня `lvl`.
- Для пары игроков `A` и `B` переводим силу в вероятность победы:
- `pA = powerA / (powerA + powerB)`
- `pB = 1 - pA`
- Победитель определяется случайным событием по этим вероятностям.

## Уточнения по дропу/даунгрейду вещей и изменению цвета

- Победитель: перед броском шанса на дроп сначала строится пул улучшений (вещи, которые лучше по статам, чем текущая вещь в этом слоте/части).
- Если пул улучшений пустой, дроп вещи невозможен (чтобы не было «выпало, но улучшить нечем»).
- Редкость сильных вещей делается не порогами, а весами: при выборе из пула улучшений вес кандидата уменьшается при росте его статов.
- Проигравший: даунгрейд вещи выбирается с учетом «силы падения».
- Чем больше `delta = (текущая_сила - новая_сила)`, тем меньше вероятность именно такого даунгрейда (слишком сильный даунгрейд маловероятен).
- Цвет после победы: не «перерандом» из распределения, а шанс апгрейда цвета ровно на 1 ступень (без даунгрейда).
- Цвет после поражения: апгрейд невозможен; есть шанс даунгрейда цвета ровно на 1 ступень (без падения на 2+ ступени).

## Матчмейкинг (чтобы не было «снежного кома» побед/поражений)

- Для каждого персонажа считаем `mmScore`.
- В качестве `mmScore` используем уже согласованную боевую силу `power`.
- На каждом тике битвы берем пул участников и перемешиваем (чтобы не было постоянных повторов).
- Сортируем участников по `mmScore`.
- Формируем пары из соседей в списке: `(0,1)`, `(2,3)`, `(4,5)` и т.д.
- Вводим порог `maxDelta`: если `abs(mmScoreA - mmScoreB) > maxDelta`, пара не создается (персонажи пропускают этот тик).
