# Components Scaling + Weight Implementation (Balance Spec)

Этот документ фиксирует однозначную спецификацию механизма `weight` (редкость компонента) и протокол расширения пула компонентов так, чтобы:
- контент можно было добавлять без "ломания" баланса,
- прогрессия была плавной (без больших прыжков силы),
- редкие вещи оставались редкими во всех контекстах (spawn / upgrade / downgrade).

## Термины и принципы

### Слоты
Компоненты персонажа разбиты на слоты: `hat`, `face`, `arms`, `torso`, `legs`.

### Статы
`hp` и `attack` начисляются суммой статов компонентов по слотам (кламп в `1..10` — как в текущей системе).

### Два разных механизма (НЕ путать)

1) Анти-скачок силы (anti-jump) в цепочке ап/даун (уже есть).
- Это про размер шага: "слегка лучше/хуже" должно быть вероятнее, чем "сразу в топ/в низ".
- Обозначения: `gain` (насколько улучшились), `drop` (насколько ухудшились).

2) Глобальная редкость компонента (`weight`).
- Это про частоту появления конкретной вещи в мире.
- Нужна, чтобы различать компоненты одинаковой силы/шага, но разной желаемой редкости/ценности/стиля.
- `weight` — абсолютный (любые положительные числа), нормализация происходит только внутри конкретного выбора.

## Поле `weight` (обязательное для каждого компонента)

- `weight` — положительное число (`> 0`).
- Чем больше `weight`, тем чаще компонент выбирается.
- Дефолт: `weight=1`.

## Где применяется `weight` (контексты выбора)

Есть три независимых контекста выбора компонента. `weight` применяем как множитель внутри каждого контекста.

### Контекст A: стартовая random-генерация (spawn)

Цель: редкость вещей сразу проявляется в генерации случайного профиля.

Алгоритм:
- Для каждого слота `slot` выбирается компонент из `randomPoolBySlot[slot]`.
- Выбор внутри слота: `P(component) ∝ component.weight`.

### Контекст B: апгрейд после победы (winner upgrade)

Цель: апгрейды происходят "по ступенькам" (anti-jump), а редкие вещи остаются редкими (weight).

Алгоритм:
- Формируем пул кандидатов: только те `componentId`, которые "лучше" текущего по вашей метрике (сейчас: `score = hp+attack`).
- Для кандидата вычисляем `gain = max(1, score(candidate) - score(current))`.
- Итоговый вес выбора кандидата:
  - `finalWeight = antiJump(gain) * rarityWeight(candidate.weight)`
  - `antiJump(gain)` — текущая функция анти-скачка (например `1/(gain^2)`).
  - `rarityWeight(w)` — базово `w` (или `w^alpha`, где `alpha≈1`).

### Контекст C: даунгрейд после поражения (loser downgrade)

Цель: даунгрейды тоже должны быть плавными (anti-jump) и учитывать редкость (weight).

Алгоритм:
- Формируем пул кандидатов: только те `componentId`, которые "хуже" текущего по метрике `score`.
- Для кандидата вычисляем `drop = max(1, score(current) - score(candidate))`.
- Итоговый вес выбора кандидата:
  - `finalWeight = antiJump(drop) * rarityWeight(candidate.weight)`

## Приоритет расширения контента (что добавлять первым)

- В первую очередь не "класс", а слоты с узким пулом: `legs` -> `face` -> `hat`.
- После этого: расширять `arms` и `torso` (там основная стат-динамика).

## Спецификация по силе новых компонентов (первый этап)

Цель: увеличить вариативность, но не поднимать резко потолки.

- `hat/face/legs`: в основном power=2, редко 1 или 3.
- `arms`: основная масса power=2-3, power=4 редкий, >4 пока не добавлять.
- `torso`: основная масса power=3-4, power=5 очень редкий.

Гайд по распределению (на слот):
- `common` (слабые/базовые): ~60-70% пула
- `mid`: ~25-35%
- `high`: ~5-10%

Важно:
- Без `weight` даже небольшое добавление "high" быстро ломает частоты выпадения (random станет давать сильные вещи слишком часто).

## Протокол внедрения (этапы без двусмысленностей)

Этап 0 (schema-only, совместимость):
- Добавить поле `weight` всем существующим компонентам, `weight=1`.
- Поведение не меняется; цель этапа — единый протокол данных.

Этап 1 (включить spawn-weight):
- Применить `weight` только в стартовой random-генерации (контекст A).
- Проверка telemetry: распределение компонентов по слотам должно соответствовать весам.

Этап 2 (включить drop-weight):
- Применить `weight` в winner upgrade и loser downgrade (контексты B/C) как множитель к anti-jump.
- Проверка telemetry: не должно появиться больших скачков силы; редкие вещи должны оставаться редкими.

Этап 3 (контент и тюнинг):
- Добавлять новые компоненты со статами и `weight`.
- Менять веса существующих компонентов только после telemetry-проверки.

## Telemetry (минимум для тюнинга веса)

Чтобы тюнить веса без "угадывания", в логах должны быть:
- контекст выбора: `spawn` / `winner_upgrade` / `loser_downgrade`
- `slot`
- `currentComponentId` (для B/C)
- размер пула кандидатов
- для выбранного кандидата: `candidateId`, `gain/drop`, `component.weight`, `finalWeight`, `roll`

## Актуальный план действий

Согласованный порядок работ:

1. Собрать baseline-статистику на минимально достаточном наборе симуляций (параметры симуляций выбираются по балансу "объем данных / стоимость прогона").
2. На основе telemetry построить гипотезу по весам текущих компонентов (без добавления новых компонентов).
3. Выполнить тюнинг `weight` уже существующих компонентов и повторно проверить гипотезу на симуляциях.
4. Только после стабилизации распределений и прогрессии переходить к добавлению новых компонентов.

Принцип: сначала стабилизируем механику выпадения/апа/дауна на текущем наборе компонентов, затем расширяем контент.
